1. Request throughput：按 case 成功请求数除以 case 总耗时计算。
2. Output / Total token throughput：分别按 sum(output_tokens) / wall_clock 和 sum(input_tokens + output_tokens) / wall_clock 计算。
3. TTFT：从发起请求到首个最终正文 token 的时间；若服务额外输出 reasoning，这里仍只统计最终回答。
4. TPOT：按 (请求总耗时 - TTFT) / max(output_tokens - 1, 1) 近似每个输出 token 的平均生成时间。
5. E2EL：单请求端到端总耗时。
6. 严格可比模式：要求接口返回流式正文 token 和准确输入/输出 token usage；OpenAI 兼容服务通常是 usage.prompt_tokens / usage.completion_tokens，Ollama Native 使用 prompt_eval_count / eval_count。
7. 目标输入档位：页面会逼近目标长度；结果优先使用接口 usage 或服务端 token API，缺失时回退启发式估算，并在状态诊断里标记。
8. 前缀缓存规避：默认会在首个 message 开头注入唯一随机前缀；识别到 vLLM 时还会尝试添加请求级 cache_salt。若需要硬保证，应在服务端直接关闭 prefix cache。
9. 轮次列：显示该 case 已完成的测量轮数；若配置轮数大于已完成轮数，会显示为 已完成/配置。CV 是多轮吞吐的变异系数（标准差 / 平均值），越低表示跨轮越稳定；单轮不显示 CV。请求明细 CSV 保留每轮原始数据。
10. 状态诊断：自动标记高抖动、部分失败、近似 token 统计、输入长度偏差和不可比请求；鼠标悬停状态胶囊可查看。
